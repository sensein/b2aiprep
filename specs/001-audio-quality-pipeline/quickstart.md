# Quickstart: Audio Quality Assurance Pipeline

**Date**: 2026-04-09
**Plan**: [plan.md](plan.md)

---

## Prerequisites

1. A BIDS dataset produced by `b2aiprep-cli redcap2bids` (or equivalent).
2. Feature `.pt` files generated by `b2aiprep-cli generate-audio-features`.
3. `b2aiprep` installed (the new QA commands will be available after this feature merges).

---

## Step 1 — Run automated QA

```bash
b2aiprep-cli qa-run /path/to/bids_dataset /path/to/qa_output
```

This runs all four checks (technical quality, unconsented speakers, PII, task compliance)
and writes:
- `audio_quality_metrics.tsv` — Stage 1 technical QC metrics per audio
- `qa_check_results.tsv` — per-check scores and classifications for every audio
- `qa_composite_scores.tsv` — composite classification per audio (pass/fail/needs_review)
- `needs_review_queue.tsv` — audios that require human review
- `qa_pipeline_config_<hash>.json` — exact config snapshot for reproducibility

To use a custom config (e.g., different thresholds or model versions):

```bash
b2aiprep-cli qa-run /path/to/bids_dataset /path/to/qa_output \
  --config my_custom_config.json
```

**Running on a SLURM cluster** (for batches > 10,000 audios): submit an array job where each
task processes a non-overlapping shard, then merge outputs:

```bash
# Example: split into 20 shards across 20 SLURM array tasks
b2aiprep-cli qa-run /path/to/bids_dataset /path/to/qa_output \
  --part $SLURM_ARRAY_TASK_ID --num-parts 20
```

If technical quality metrics are already computed from a prior run, skip Stage 1:

```bash
b2aiprep-cli qa-run /path/to/bids_dataset /path/to/qa_output --use-existing-qc
```

---

## Step 2 — Human review of flagged audios

```bash
b2aiprep-cli qa-review /path/to/qa_output --reviewer-id jsmith \
  --audio-root /path/to/bids_dataset
```

For each flagged audio you will see the per-check confidence breakdown and be prompted
to accept or reject. Decisions are saved to `human_review_decisions.tsv` and the session
is resumable — press `[q]` at any time and re-run the command to continue where you left off.

On HPC nodes without audio output, omit `--audio-root` to review scores only:

```bash
b2aiprep-cli qa-review /path/to/qa_output --reviewer-id jsmith
```

---

## Step 3 — Generate the release report

```bash
b2aiprep-cli qa-report /path/to/qa_output --confidence-target 0.95
```

Generates `qa_release_report.md` and `qa_release_report.json` with per-check pass rates
and a top-level confidence claim about the batch.

To fail the command (for CI gating) if the dataset does not meet the confidence target:

```bash
b2aiprep-cli qa-report /path/to/qa_output \
  --confidence-target 0.95 --fail-on-below-target
echo $?  # 0 = passed, 2 = below target
```

## Validation

Run the test suite to verify the pipeline works end-to-end on synthetic data:

```bash
pytest tests/test_task_compliance.py \
       tests/test_pii_detection.py \
       tests/test_unconsented_speakers.py \
       tests/test_qa_report.py \
       -v
```

## Reproducing a prior run

Every run records the config file used. To reproduce:

```bash
b2aiprep-cli qa-run /path/to/bids_dataset /path/to/new_output \
  --config /path/to/qa_output/qa_pipeline_config_a3f7c891.json
```

Providing the same config file and the same BIDS input must produce identical
`qa_composite_scores.tsv` output (constitution Principle I: Reproducibility).
